## 2025-02-17 - Asynchronous File I/O Optimization in FeedbackService

**Learning:** The Express backend orchestration server was utilizing synchronous blocking file operations (`fs.readFileSync`) within `FeedbackService.triggerProcessor` to read multiple feedback files (metadata, agent prompt, screenshot) sequentially. In high-concurrency Node.js environments, blocking the event loop with synchronous operations introduces severe scalability and performance bottlenecks, directly impacting API response times and throughput. However, since Vercel's edge/serverless filesystem restricts `fs.promises` functionality or behaves differently, standardizing on asynchronous reads without fallback can lead to crashes if `fs.promises` is undefined. A resilient approach is necessary to ensure both optimization and backward compatibility.
**Action:** Replaced sequential `fs.readFileSync` calls with parallelized, non-blocking asynchronous operations using `Promise.all` alongside `fs.promises.readFile`. Added fallback mechanisms to gracefully revert to asynchronous-wrapped `fs.readFileSync` if `fs.promises` is unavailable, maintaining identical logic flow for optional files (by ignoring `ENOENT` errors). This guarantees that the event loop remains unblocked, substantially enhancing endpoint concurrency and execution speed without introducing regressions in specialized deployment environments.
